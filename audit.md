## ⚠️ ПОТЕНЦИАЛЬНЫЕ ПРОБЛЕМЫ И ЗАМЕЧАНИЯ

### 🔴 КРИТИЧЕСКИЕ

1. **Отсутствие rate-limiting на уровне эндпоинта**
   ```typescript
   // ❌ Нет проверки на частые повторные запуски
   @Post('scan/start')
   startScan(@Body() body: ...) {
     return this.domainScannerService.startScan(body);
   }
   ```
   **Проблема:** Если пользователь быстро нажмёт кнопку несколько раз, первый запрос вернёт результат, но другие получат 429. Фронтенд перехватывает это, но это не оптимально.  
   **Рекомендация:** Добавить флаг в frontend для отключения кнопки сразу при click.

2. **Отсутствие таймаута на фронтенде**
   ```typescript
   // ❌ Бесконечный polling может продолжаться при разрыве соединения
   const tick = async () => {
     const status = await fetchScanStatus();
     // ... нет таймаута на останов polling
   };
   const timer = window.setInterval(tick, 1000);
   ```
   **Проблема:** Если сервер зависнет, polling будет продолжаться.  
   **Рекомендация:** Добавить максимум попыток или реал-таймаут (e.g., 15 минут).

3. **Memory leak при ошибке в polling**
   ```typescript
   } catch (e) {
     if (!cancelled) {
       console.error(e);  // ❌ Может создать множество логов
     }
   }
   ```
   **Проблема:** Нет обработки повторных ошибок в цикле.

### 🟡 УМЕРЕННЫЕ

4. **Недостаточная валидация на backend перед spawn**
   ```typescript
   const args = [
     '--signal=TERM',
     `${scanSeconds}s`,
     this.scannerBin,
     '--addr', addr,  // ❌ addr не выполняет shell-escaping
   ];
   ```
   **Проблема:** Если `addr` содержит специальные символы, может быть уязвимость.  
   **Проверка:** spawn используется без shell (stdio: ['ignore', 'pipe', 'pipe']), поэтому это **не критично**, но стоит добавить явную валидацию addr:
   ```typescript
   const isValidAddr = /^[a-zA-Z0-9\.\-\:]+$/.test(addr);
   if (!isValidAddr) throw new BadRequestException('Некорректный адрес');
   ```

5. **Потеря состояния при перезагрузке контейнера**
   - `activeScan` и `lastScanResult` хранятся в памяти
   - Если контейнер рестартит, состояние теряется
   - **Это нормально для MVP**, но стоит задокументировать

6. **Жёсткий размер лога (8000 символов)**
   ```typescript
   private readonly logTailLimit = 8000;
   ```
   **Проблема:** При очень большом количестве доменов логи могут быть обрезаны.  
   **Вероятность проблемы:** Низкая, но стоит мониторить.

### 🟢 МИНОРНЫЕ

7. **BOM-символ в импорте доменов**
   ```typescript
   let value = (input || '').replace(/^\uFEFF/, '').trim();
   ```
   ✅ Хорошо, что обработано.

8. **Отсутствие документации для API**
   - Нет JSDoc комментариев для методов scanner
   - Рекомендация: добавить хотя бы примеры использования

9. **Жёсткие диапазоны параметров**
   ```typescript
   const scanSeconds = this.clampNumber(payload.scanSeconds, 120, 10, 600);
   const thread = this.clampNumber(payload.thread, 2, 1, 20);
   ```
   Эти значения захардкодены. Стоит вынести в env переменные для prod-конфигурирования.

10. **Дублирование кода в nginx.conf**
    - Один и тот же блок timeouts дублируется для двух серверов
    - Меньше критично, но нарушает DRY принцип

## 🧪 ТЕСТИРОВАНИЕ

### ✅ Что проверено в PR
- старт/остановка скана
- удаление отдельных записей
- импорт найденных доменов
- восстановление после F5
- сложные входные данные (комментарии, wildcard)
- экспорт списков

### ❓ Что не проверено / нужны тесты
1. **Граничные случаи:**
   - Пустой результат сканирования
   - Сканирование на несуществующий адрес
   - Очень длинный список доменов (1000+)
   - Символы Unicode в названиях хостов

2. **Производительность:**
   - Поведение с 10к+ доменов в whitelist
   - Загрузка localStorage при большом размере

3. **Безопасность:**
   - SQL injection (маловероятна благодаря TypeORM, но стоит проверить)
   - XSS через domain names (должны быть экранированы, нужна проверка)
