## Что реализовано корректно

### 1) Архитектура и интеграция
- Сканирование вынесено в backend (контейнер), что правильно для low-level TLS/сетевого поведения.
- Frontend получил полноценный сценарий: запуск скана, предварительный список, ручная отбраковка, импорт в основной whitelist.
- Реализована условная видимость блока сканера: показывается только при доступном бинарнике.

### 2) Backend scanner
- Есть защита от параллельных запусков (один активный скан на инстанс).
- Корректно обрабатываются коды завершения timeout (`124/137/143`) как ожидаемый сценарий.
- Реализован парсинг доменов из потокового вывода с учетом разрезанных чанков.
- Добавлены новые endpoint’ы статуса:
  - `GET /api/domains/scan/status`
  - `GET /api/domains/scan/last-result`
- Это дает восстановление UX после F5 и backend-driven статус.

### 3) Frontend UX
- Добавлен polling статуса скана раз в 1 сек.
- Таймер теперь идет от backend-статуса, а не от локальной оценки.
- После перезагрузки страницы UI подхватывает активный скан.
- При `429` (скан уже идет) UI подключается к текущему запуску вместо “падения”.
- Добавлены экспорт/очистка/ручная проверка ссылками.

### 4) Валидация доменов
- Нормализация и фильтрация импорта/ручного ввода усилены.
- Учитываются комментарии, wildcard, URL-обертки, порты, дубликаты без учета регистра.
- Валидация доменных частей аккуратная и практичная для текущего кейса.

### 5) DevOps/операционка
- RealiTLScanner собирается reproducible способом (pin commit) в multi-stage build.
- Добавлены runtime-утилиты в backend-образ для сценария сканирования.
- Исправлен 504 на длинных сканах: увеличены timeout’ы прокси `/api/`.
- Улучшена совместимость install/update: поддержка `docker compose` (v2) и `docker-compose` (v1).

## Что в изначальном Copilot-ревью было неточно

1. **“Нет защиты от частых кликов на запуск”** — неверно.  
   На frontend кнопка блокируется в состоянии скана, на backend есть защита и ответ `429`.

2. **“Memory leak из-за polling”** — неверно.  
   Утечки памяти не обнаружено. Реальный риск — шумные логи при длительной недоступности API.

3. **“Command injection через addr” как критическая угроза** — переоценено.  
   `spawn` без shell значительно снижает риск инъекции команд.  
   Но форматная валидация `addr` всё равно желательна как hardening.

## Реальные риски / что стоит доработать

### P1 (желательно)
1. Явная валидация `addr` на backend (hostname/IP) до запуска scanner.

### P2 (рекомендуется)
1. Добавить backoff/ограничение при повторных ошибках polling (чтобы не засорять логи).
2. Коротко задокументировать scanner API в README/документации.

### P3 (nice-to-have)
1. Вынести диапазоны `scanSeconds/thread/timeout` в конфиг/env.
2. Если нужен recovery после перезапуска контейнера — хранить status/result не только в памяти.

## Проверка по безопасности

| Вектор | Статус | Комментарий |
|---|---|---|
| Command injection | 🟡 | Существенно снижен за счет `spawn` без shell; добавить валидацию `addr` для hardening |

| DoS | ✅ | Ограничение одного активного скана на инстанс |
| Long request timeout | ✅ | Прокси timeout для `/api/` увеличены |
